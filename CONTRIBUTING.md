# Contributing to holoctl

Thanks for considering a contribution! holoctl is small and easy to hack on. This file is a tour of how the project is laid out, how to run it locally, and the conventions to follow.

For high-level design see [ARCHITECTURE.md](ARCHITECTURE.md).

## TL;DR

```bash
git clone https://github.com/FelipeCarillo/holoctl
cd holoctl

# Python implementation (canonical)
uv sync
uv run holoctl --help
uv run pytest tests/         # if/when tests exist

# Node implementation (mirror)
npm install
node bin/holoctl.js --help
npm test                      # node --test src/**/*.test.js
```

Make your change, run the smoke test (below), open a PR.

## Project layout

```
.
├── bin/holoctl.js         Node CLI entrypoint
├── src/                   Node implementation
├── holoctl/               Python implementation (canonical for PyPI)
├── pyproject.toml         Python pkg metadata
├── package.json           Node pkg metadata
├── ARCHITECTURE.md        internal design
├── SECURITY.md            vulnerability reporting
└── .holoctl/              dogfood — this repo uses holoctl on itself
```

## Dual-stack rule

holoctl ships **two parallel implementations** (Node + Python) of the same CLI. When you change behavior in one, mirror it in the other or call it out explicitly in the PR.

Why not pick one? Historical reason — Node came first, then a Python rewrite for PyPI distribution. Deprecating one is on the roadmap; until then, parity matters.

| Concern | Node file | Python file |
|---|---|---|
| Workspace location | `src/lib/config.js` | `holoctl/lib/config.py` |
| Subproject discovery | `src/lib/discover.js` | `holoctl/lib/discover.py` |
| Board/ticket logic | `src/lib/board.js` | `holoctl/lib/board.py` |
| Compile target | `src/lib/compiler/<target>.js` | `holoctl/lib/compiler/<target>.py` |
| Templates | `src/templates/index.js` | `holoctl/lib/templates.py` |
| Web dashboard | `src/server/` (Hono) | `holoctl/server/app.py` (FastAPI) |

## Local dev setup

### Python (canonical)

Requires Python ≥ 3.11 and [uv](https://docs.astral.sh/uv/) (or `pip` + venv if you prefer).

```bash
uv sync                              # creates .venv/ and installs holoctl + deps editable
uv run holoctl init                  # try it on dev-sandbox/
uv run holoctl serve                 # http://127.0.0.1:4242
```

Run any source change and re-invoke `uv run holoctl …` — editable install picks up changes immediately.

### Node (mirror)

Requires Node ≥ 18.

```bash
npm install
node bin/holoctl.js init
node bin/holoctl.js serve
```

### Sandbox for safe experiments

`dev-sandbox/` at the repo root is gitignored. Use it to test `init`, discovery, multi-project flows without polluting the repo or `$HOME`:

```bash
mkdir dev-sandbox/scratch
cd dev-sandbox/scratch
uv run --project ../.. holoctl init
mkdir app && (cd app && npm init -y)
mkdir api && touch api/pyproject.toml
uv run --project ../.. holoctl overview
```

## Smoke test before opening a PR

There's no formal CI yet for end-to-end behavior, so run this manually after non-trivial changes:

```bash
# Python
rm -rf dev-sandbox/_smoke && mkdir dev-sandbox/_smoke && cd dev-sandbox/_smoke
uv run --project ../.. holoctl init --name smoke --prefix SM --skip-compile
mkdir app api && echo '{}' > app/package.json && echo '[project]' > api/pyproject.toml
uv run --project ../.. holoctl overview         # expect: 📁 Projects with app + api
uv run --project ../.. holoctl board add '{"title":"x","projects":["app"]}'
uv run --project ../.. holoctl board ls --project app  # expect: SM-001 listed

# Node parity (run all of the above with `node ../../bin/holoctl.js` instead)
```

If both stacks behave the same, you're good.

## Conventions

### Code

- **No comments unless they explain a non-obvious WHY.** Identifiers should already say WHAT.
- **Don't add error handling for impossible cases.** Trust internal call sites; validate only at boundaries (CLI args, file I/O, parsing user content).
- **Don't introduce abstractions for hypothetical futures.** Three similar lines beat a premature helper.

### File generation

Anything generated by `holoctl compile` must start with:

```html
<!-- Generated by holoctl. Do not edit directly. Source: .holoctl/ -->
```

Users will trust this banner — don't break it.

### Schema changes (board / config / agent)

Breaking a field is a **major bump**. Adding a field with a sensible default is a minor bump. Tickets in the wild keep working because:

1. `board.rebuild_index` reads the old field and the new field, prefers the new one (with fallback).
2. The next write uses the new field exclusively.

When you add a field, mirror that pattern — see how `scope: string` → `projects: string[]` was migrated in 0.5.0 ([CHANGELOG](CHANGELOG.md)).

### Templates and `sync`

If you add a new file generated by `init` from a template (anything under `.holoctl/commands/`, `WORKFLOW.md`, `_template.md`), add its path to `SYNC_TARGETS` in `src/cli/commands/sync.js` so users get it on the next `holoctl sync` after upgrading.

### Compile targets

Same target name (`claude`, `cursor`, `windsurf`, `copilot`, `devin`, `generic`) must exist in both implementations. Adding a new one means:

1. New module: `src/lib/compiler/<name>.js` + `holoctl/lib/compiler/<name>.py`.
2. Register it in the dispatch (in `compiler/index.js` Node; in `compiler/__init__.py` Python).
3. New template files under `src/templates/commands/<name>-*.md` and `holoctl/templates/commands/<name>-*.md` if the target supports a slash command.
4. Document it in [README.md](README.md) "Pick your AI tool" table and in [ARCHITECTURE.md](ARCHITECTURE.md) compile pipeline.

## Versioning + release

Semantic versioning. Both `package.json` and `pyproject.toml` carry the same version — bump them together.

Currently only PyPI is published. Release ritual:

```bash
# 1. Bump version in package.json + pyproject.toml + CHANGELOG.md
# 2. Build
rm -rf dist && uv build
# 3. Validate
uv tool run twine check dist/*
# 4. Test the wheel in a clean venv
uv venv /tmp/check && uv pip install --python /tmp/check/Scripts/python.exe dist/holoctl-*.whl
/tmp/check/Scripts/holoctl --version
# 5. Publish (uses ~/.pypirc)
uv tool run twine upload dist/holoctl-*.whl dist/holoctl-*.tar.gz
```
