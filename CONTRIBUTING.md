# Contributing to holoctl

Thanks for considering a contribution! holoctl is small and easy to hack on. This file is a tour of how the project is laid out, how to run it locally, and the conventions to follow.

For high-level design see [ARCHITECTURE.md](ARCHITECTURE.md).

## TL;DR

```bash
git clone https://github.com/FelipeCarillo/holoctl
cd holoctl

uv sync
uv run holoctl --help
uv run pytest tests/
```

Make your change, run the smoke test (below), open a PR.

## Project layout

```
.
├── holoctl/               Python package (canonical)
├── tests/                 pytest suite
├── pyproject.toml         package metadata
├── ARCHITECTURE.md        internal design
├── SECURITY.md            vulnerability reporting
└── .holoctl/              dogfood — this repo uses holoctl on itself
```

| Concern | File |
|---|---|
| Workspace location | `holoctl/lib/config.py` |
| Subproject discovery | `holoctl/lib/discover.py` |
| Board/ticket logic | `holoctl/lib/board.py` |
| Compile target | `holoctl/lib/compiler/<target>.py` |
| /holoctl bootstrap per target | `holoctl/templates/commands/holoctl-<target>.md` |
| Init-time templates | `holoctl/lib/templates.py` |
| Web dashboard | `holoctl/server/app.py` |

## Local dev setup

Requires Python ≥ 3.11 and [uv](https://docs.astral.sh/uv/) (or `pip` + venv if you prefer).

```bash
uv sync                              # creates .venv/ and installs holoctl + deps editable
uv run holoctl init                  # try it on dev-sandbox/
uv run holoctl serve                 # http://127.0.0.1:4242
```

Run any source change and re-invoke `uv run holoctl …` — editable install picks up changes immediately.

### Sandbox for safe experiments

`dev-sandbox/` at the repo root is gitignored. Use it to test `init`, discovery, multi-project flows without polluting the repo or `$HOME`:

```bash
mkdir dev-sandbox/scratch
cd dev-sandbox/scratch
uv run --project ../.. holoctl init
mkdir app && (cd app && npm init -y)
mkdir api && touch api/pyproject.toml
uv run --project ../.. holoctl overview
```

## Smoke test before opening a PR

```bash
rm -rf dev-sandbox/_smoke && mkdir dev-sandbox/_smoke && cd dev-sandbox/_smoke
uv run --project ../.. holoctl init --name smoke --prefix SM --skip-compile
mkdir app api && echo '{}' > app/package.json && echo '[project]' > api/pyproject.toml
uv run --project ../.. holoctl overview         # expect: 📁 Projects with app + api
uv run --project ../.. holoctl board add '{"title":"x","projects":["app"]}'
uv run --project ../.. holoctl board ls --project app  # expect: SM-001 listed
```

Plus `uv run pytest` for the unit suite.

## Conventions

### Code

- **No comments unless they explain a non-obvious WHY.** Identifiers should already say WHAT.
- **Don't add error handling for impossible cases.** Trust internal call sites; validate only at boundaries (CLI args, file I/O, parsing user content).
- **Don't introduce abstractions for hypothetical futures.** Three similar lines beat a premature helper.

### File generation

Anything generated by `holoctl compile` must start with:

```html
<!-- Generated by holoctl. Do not edit directly. Source: .holoctl/ -->
```

Users will trust this banner — don't break it.

### Schema changes (board / config / agent)

Breaking a field is a **major bump**. Adding a field with a sensible default is a minor bump. Tickets in the wild keep working because:

1. `board.rebuild_index` reads the old field and the new field, prefers the new one (with fallback).
2. The next write uses the new field exclusively.

When you add a field, mirror that pattern — see how `scope: string` → `projects: string[]` was migrated in 0.5.0 ([CHANGELOG](holoctl/CHANGELOG.md)).

### Templates and `sync`

If you add a new file generated by `init` from a template (anything under `.holoctl/commands/`, `WORKFLOW.md`, `_template.md`), add its path to `_SYNC_TARGETS` in `holoctl/cli/sync_.py` so users get it on the next `holoctl sync` after upgrading.

### Adding a compile target

1. New module: `holoctl/lib/compiler/<name>.py`. Function signature: `def compile_<name>(project_root: Path, config: dict, dry_run: bool = False) -> dict`.
2. Register it in the dispatch dict in `holoctl/lib/compiler/__init__.py`.
3. If the target supports a slash command for `/holoctl`, add the template at `holoctl/templates/commands/holoctl-<name>.md` and load it via `compiler.template.load_bootstrap(...)`.
4. Document it in [README.md](README.md) "Pick your AI tool" table and in [ARCHITECTURE.md](ARCHITECTURE.md) compile pipeline.
5. Add the expected output paths to `_TARGET_OUTPUTS` in `holoctl/cli/doctor.py`.

## Versioning + release

Semantic versioning. The version in `pyproject.toml` and `holoctl/__init__.py` (fallback) should match the latest CHANGELOG entry.

### Preferred: tag-driven release via GitHub Actions (OIDC)

Since 0.15.0 the `release.yml` workflow handles publishing automatically when a `v*` tag is pushed. **One-time setup**: configure a [PyPI Trusted Publisher](https://docs.pypi.org/trusted-publishers/) at <https://pypi.org/manage/account/publishing/> pointing to:

| Field | Value |
|---|---|
| PyPI Project Name | `holoctl` |
| Owner | `FelipeCarillo` |
| Repository name | `holoctl` |
| Workflow name | `release.yml` |
| Environment name | `pypi` |

After that, the release ritual is:

```bash
# 1. Bump version in pyproject.toml + holoctl/__init__.py fallback + holoctl/CHANGELOG.md
# 2. Commit + open PR + merge to main
# 3. Tag the merge commit and push
git tag v0.15.0
git push --tags
```

The workflow:

1. Re-runs the full test matrix on the exact tagged commit (3.11/3.12/3.13 × Ubuntu/macOS/Windows).
2. Verifies the tag matches `pyproject.toml`'s `version`.
3. `uv build` → wheel + sdist.
4. `twine check` validates metadata.
5. **`pypa/gh-action-pypi-publish` uploads via OIDC** — no API token in repo secrets.
6. Creates a GitHub Release with notes extracted from the matching `## [VERSION]` slice in `holoctl/CHANGELOG.md`, attaching the wheel + sdist.

`skip-existing: true` makes re-runs idempotent if a partial failure leaves only some artifacts uploaded.

### Fallback: manual release via twine

If the trusted publisher is misconfigured or the workflow is disabled, the legacy ritual still works (uses `~/.pypirc` with an API token):

```bash
rm -rf dist && uv build
uv tool run twine check dist/*

# Test the wheel in a clean venv
uv venv /tmp/check && uv pip install --python /tmp/check/bin/python dist/holoctl-*.whl
/tmp/check/bin/holoctl --version

# Publish
uv tool run twine upload dist/holoctl-*.whl dist/holoctl-*.tar.gz
```
