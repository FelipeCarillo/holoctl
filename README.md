Type `/holoctl` in any AI coding assistant. The agent reads the workspace state, plants the structure if needed, and routes itself to the right flow — first time, upgrade, or normal operation.

Works in **Claude Code**, **Cursor**, **Windsurf**, **GitHub Copilot**, **Devin**, **Aider**, and any agent that reads `AGENTS.md` / `CLAUDE.md`.

```bash
uv tool install holoctl
hctl setup                    # one-time per machine — plants /holoctl in every detected assistant
```

Now, in any folder:

```bash
hctl init                     # creates .holoctl/, compiles for every target, plants hooks + MCP
```

Or just type `/holoctl` inside the assistant — it runs `hctl init` itself.

---

## Why "living"

`holoctl` started as a static compiler — `.holoctl/` to native files. As of **0.14** it's a system that wakes up between sessions:

- **Durable memory** at `.holoctl/memory/` shared across all assistants — the same notes show up in Claude, Cursor, Windsurf, Copilot, and Devin in each one's native format.
- **Event journal** captures every tool use, edit, and session boundary via hooks plumbed automatically by `hctl init`.
- **Autonomous curator** watches the journal and proposes new personas, path-scoped rules, or topic archives as `meta:curate` tickets on the board. Approve a suggestion by moving the ticket to `done` — it auto-executes the proposed action.
- **Token-economy boot** prints ≤1KB of session-zero context (top pendings, recent decisions, available topics) so the assistant doesn't burn tokens loading the whole `CLAUDE.md`.
- **MCP server** exposes the board, memory, journal, and curator as standard tools — read/write with permission gating in Claude Code.

```
your-project/
├── .holoctl/                       ← single source of truth, committed to git
│   ├── config.json
│   ├── instructions.md             ← compiled to CLAUDE.md / AGENTS.md / etc.
│   ├── board/                      ← Kanban + tickets
│   │   ├── index.json              ← auto-rebuilt from ticket .md files
│   │   └── tickets/PRJ-001-*.md
│   ├── agents/                     ← active personas (only `boardmaster` by default)
│   ├── commands/                   ← /board, /ticket, /sprint, /close, /decision, /status
│   ├── context/decisions/          ← ADR-style hard locks
│   ├── memory/                     ← durable cross-assistant notes
│   │   ├── MEMORY.md               ← always-on index
│   │   └── topics/                 ← lazy/glob-scoped topics
│   ├── journal/                    ← daily JSONL of session events
│   └── curator/                    ← curator state + per-ticket metadata
└── …your code
```

Compiled outputs (`CLAUDE.md`, `.cursor/rules/`, `.windsurf/skills/`, `.vscode/mcp.json`, `.devin/skills/`, etc.) are regenerated by `hctl init` and `hctl compile`. Most users `.gitignore` them.

## Install

**Requires Python ≥ 3.11.**

```bash
uv tool install holoctl              # recommended — venv isolation, hctl on PATH
# or
pipx install holoctl
# or
pip install holoctl                  # inside an active venv

# Optional ML extra (curator paraphrase detection — ~250MB, ONNX backend):
uv tool install "holoctl[ml]"
```

> **`holoctl: command not found`?** `uv tool` and `pipx` put the CLI on PATH for you. With plain `pip`, add `~/.local/bin` (Linux/Mac) or `~/AppData/Roaming/Python/Scripts` (Windows) to your PATH, or call `python -m holoctl`.

## Setup once, work in any folder

```bash
hctl setup
```

`hctl setup` detects every supported AI assistant on your machine and plants the `/holoctl` skill in **user scope**:

```
✓ Claude Code   → ~/.claude/commands/holoctl.md
✓ Cursor        → ~/.cursor/rules/holoctl.mdc
✓ Windsurf      → ~/.codeium/windsurf/skills/holoctl/SKILL.md
✓ Copilot       → ~/.copilot/prompts/holoctl.prompt.md
✓ Devin CLI     → ~/.config/devin/skills/holoctl/SKILL.md
```

The skill body is identical across all 5 — only the frontmatter changes. From now on, in any folder, you type `/holoctl` and the agent runs `hctl doctor`, picks the right flow, and gets to work:

| `hctl doctor` says | The agent runs |
|---|---|
| `not initialized` | `hctl init` (creates `.holoctl/`, compiles, plants hooks + MCP) |
| `outdated` | `hctl upgrade --check` then asks before applying |
| `ok` | `hctl boot` (≤1KB session-zero context) |

## Five assistants, one source of truth

`.holoctl/` compiles to each target's **native primitive** — no monkey-patch:

| Assistant | Memory | Skills | Hooks | MCP |
|---|---|---|---|---|
| Claude Code | `.claude/skills/holoctl-memory*/SKILL.md` (lazy, glob, always_on) | `.claude/skills/<name>/SKILL.md` | `.claude/settings.json:hooks` | `mcpServers.holoctl` |
| Cursor | `.cursor/rules/holoctl-memory*.mdc` (alwaysApply / globs / description) | `.cursor/rules/skill-*.mdc` | `.cursor/hooks.json` | `.cursor/mcp.json` |
| Windsurf | `.windsurf/rules/holoctl-memory*.md` (`trigger: always_on \| model_decision \| glob`) | `.windsurf/skills/<name>/SKILL.md` | — | `.windsurf/mcp.json` |
| Copilot | `.github/instructions/holoctl-memory-*.instructions.md` (`applyTo:`) | `.github/prompts/*.prompt.md` | — | `.vscode/mcp.json` |
| Devin | `.devin/rules/holoctl-memory*.md` | `.devin/skills/<name>/SKILL.md` (also reads `.windsurf/skills/`) | — | `.devin/mcp.json` |

**Adding a memory topic is one file** in `.holoctl/memory/topics/` with `scope: lazy/glob/always_on` — the 5 compilers translate to the right native shape. Same for personas: one `.md` in `holoctl/templates/agents/` (the latent library) is opt-in via `hctl agent add <name>` and lands as a `SKILL.md` in 3 assistants natively, with `.mdc` and `.prompt.md` translations for the other two.

## What you get

### 📋 Kanban board with parallel-safe batch creation

```bash
hctl board add '{"title":"Add JWT auth","agent":"developer","priority":"p1"}'
hctl board batch '{"shared":{"tags":["par:auth"]},"tickets":[…]}'   # validates non-overlap
hctl board ls --status doing --priority p1
hctl board move PRJ-001 doing
```

### 🧠 Durable cross-assistant memory

```bash
hctl memory add decisions --scope lazy -d "Architectural decisions log"
hctl memory add api-conventions --scope glob -g "src/api/**"
hctl memory list                    # active topics + scopes
hctl memory search "JWT"
hctl memory archive old-topic
```

Topics declare canonical frontmatter (`scope`, `globs`, `description`); compilers translate per assistant.

### 🤖 Latent persona library + opt-in activation

By default `hctl init` materializes only `boardmaster`. The full catalog (`developer`, `reviewer`, `architect`, `researcher`, …) lives latent in `holoctl/templates/agents/` and activates on demand:

```bash
hctl agent list                     # active vs library
hctl agent add developer            # materialize from library with placeholders resolved
hctl agent remove developer
```

Each library persona declares `when_to_suggest:` heuristics that the curator uses to propose activations automatically.

### 📓 Event journal + curator suggestions

Hooks plumbed by `hctl init` write every session event to `.holoctl/journal/<YYYY-MM-DD>.jsonl`. The curator reads it and proposes:

| Pattern | Action proposed | Materializes as |
|---|---|---|
| ≥10 edits in `src/api/**` over 3 sessions | `rule_extract` | memory topic with `scope: glob` |
| Repeated similar prompts (≥3x) | `memory_promote` | lazy memory topic |
| Library persona heuristic fires | `agent_add` | active persona in `.holoctl/agents/` |
| Topic untouched ≥60 days | `topic_archive` | move to `topics/_archived/` |
| Cascade auto-memory ≥7 days old | `memory_promote` | versioned topic |

```bash
hctl curate run --auto              # rate-limited (1/day, 14-day silence)
hctl curate show                    # open meta:curate tickets
hctl board move PRJ-042 done        # ← approval auto-executes the action
hctl curate silence <pattern_id>    # 14-day suppression
```

### 🚀 Boot + handoff for cross-session continuity

```bash
hctl boot                           # ≤1KB: pendings, decisions, topics, personas, ⚡ curator hints
hctl handoff --note "shipped 0.14"  # appends 1 line to memory/topics/session-trail.md
```

### 🔌 MCP server — board/memory accessible from any assistant

`hctl serve --mcp` runs as a stdio MCP server; `hctl init` writes the per-assistant config so the assistant spawns it on demand. **14 tools** split read/write — write tools (`board_create`, `board_move`, `memory_add`, `agent_add`, …) land in `permissions.ask` for Claude Code. Schema mirrors the CLI 1:1.

### 🌐 Live web dashboard

```bash
hctl serve                          # http://127.0.0.1:4242
```

| Tab | Content |
|---|---|
| **Board** | Three views — **Kanban**, **List**, **Timeline** — with real-time SSE updates, inline create/move/edit, multi-select + bulk actions, search + chip filters, sort/group |
| **Repos** | Auto-discovered subprojects, git branch + ticket counts |
| **Agents** | Active personas + library catalog |
| **Commands** | Slash command library |
| **Context** | Decisions log + free-form documents |

**Board views** (`?view=kanban|list|timeline`):

- **Kanban** — column per status, cards with priority dot, agent avatars, sprint pill, due date, body preview. Inline `+ Add ticket` per column; hover ⋯ menu for Move / Archive.
- **List** — dense table grouped by status (or any axis from Sprint/Agent/Tag), sticky group headers, multi-select with Shift+Click range, sticky bulk-action bar (Move to · Archive), inline edit on status/priority cells.
- **Timeline** — horizontal roadmap with sprints (or agents) as lanes, bars spanning `created → completed||today`, today marker, three zoom levels (Week / Month / Quarter).
- **Detail page** — large header (priority dot + ID + status pill + priority pill + title), markdown body on the left, three info cards on the right (Properties / Linked / Activity) with inline-editable fields and reverse-scan `blocks` links.

All board mutations also speak HTTP:

```text
POST   /api/project/{alias}/tickets             create       { title, status?, priority?, agent?, sprint?, tags? }
POST   /api/project/{alias}/tickets/{id}/move   move         { status }
PATCH  /api/project/{alias}/tickets/{id}        edit field   { field, value }
GET    /api/project/{alias}/board-html          fragment for kanban SSE swap
GET    /api/project/{alias}/list-html           fragment for list SSE swap
GET    /api/project/{alias}/timeline-html       fragment for timeline SSE swap
```

## Commands

```
hctl setup                    Plant /holoctl skill in every detected assistant (one-time)
hctl init                     Initialize or sync .holoctl/ — idempotent
hctl upgrade                  Migrate workspace + recompile to installed version
hctl compile                  Generate AI-tool integration files
hctl serve [--mcp]            Web dashboard, or stdio MCP server
hctl doctor                   Health check (not initialized | outdated | ok)
hctl overview                 One-screen workspace snapshot

hctl board <cmd>              Tickets — add, ls, move, set, batch, stat, get, body, rebuild-index
hctl agent <cmd>              Personas — list, add, remove
hctl memory <cmd>             Memory — list, add, get, search, archive, seed
hctl journal <cmd>            Journal — record, show, count, import, tail
hctl boot                     Minimal session-zero context (≤1KB)
hctl handoff [--note ...]     End-of-session: append session-trail line
hctl curate <cmd>             Curator — run, show, silence, apply

hctl repo <cmd>               Discovered subprojects — list, add, info
```

Every command supports `--help`.

## Configuration

`.holoctl/config.json` — only override what you need:

```json
{
  "holoctlVersion": "0.14.0",
  "project": {
    "name": "My Project",
    "prefix": "MP"
  },
  "board": {
    "statuses": ["backlog", "doing", "review", "done", "cancelled"],
    "priorities": ["p0", "p1", "p2", "p3"],
    "idPadding": 3
  },
  "git": { "checkDirty": false },
  "targets": ["claude", "cursor", "windsurf", "copilot", "devin"],
  "server": { "port": 4242, "theme": "dark" }
}
```

`git.checkDirty` defaults to **false** — holoctl reads `.git/HEAD`/`refs`/`config` directly without spawning subprocesses, instant on Windows + corporate AV.

## Privacy and coexistence

- **`holoctl` writes nothing to `$HOME` from `init`.** `hctl setup` does, but only the `/holoctl` skill files in user-scope locations of detected assistants. No machine-wide registry, no daemon.
- **`.holoctl/memory/.gitignore`** is created with `_archived/` excluded by default. Privacy-strict workspaces uncomment two lines to make the whole memory tree local-only.
- **Coexists with native auto-memory.** Claude Code's auto-memory is NOT disabled — `holoctl` adds a `@.holoctl/memory/MEMORY.md` reference to `CLAUDE.md` so Claude reads both. Same with Windsurf: we write to `.windsurf/rules/` (durable, versioned), Cascade keeps its `~/.codeium/windsurf/memories/` (machine-local). The curator can promote a long-lived Cascade memory to a versioned topic if it survives ≥7 days.

## Migrating from `projctl` / `projhub`

Earlier names of this project. holoctl reads `.projctl/` and `.projhub/` directories and auto-renames them to `.holoctl/` on the next save. Tickets that used `scope: X` are read as `projects: [X]` and rewritten on the next `board set` or `rebuild-index`.
